## A. 繰り返し遊べる要素（レビュー推奨・優先）

### A-1. 実績システム 【推奨・最初に着手】
- 例：Moon Master / Kaos Expert / Air Collector、初勝利、○連勝、逆転勝ち、無傷勝ち、Kaos0で5を食う、など
- **完全ローカルで実装可能**（バックエンド不要）。既存の対戦履歴（localStorage `maskd_game_history_v1`）と1試合サマリーから判定できる
- 実装メモ：解除条件を関数群で定義 → 試合終了時（endGame）と通算集計時にチェック → localStorageに解除状態を保存 → 一覧モーダルで表示。新規解除時は演出を出す
- リスク低・即効性高

### A-2. デイリーチャレンジ
- 「全員が同じ配牌」を**日付シードの決定論的乱数**で実現（例：seed = YYYYMMDD）
- 現在の `shuffle()` / `dealHands()` / `pick6()` は `Math.random()` 依存 → **シード可能なPRNG（mulberry32等）に差し替え可能な形**にするのが肝
- 自己完結。将来の「リプレイ／結果共有」の土台にもなる
- 実装メモ：シードRNGを注入できるよう乱数を一本化 → デイリー用の固定シードで配札 → その日のスコアをローカル保存＆共有テキスト生成

### A-3. レート / ランクマッチ
- 対CPUなので実質「難易度別の自己ELO」。勝敗でレート増減、難易度で係数変更
- ローカルで実装可能。ランキング化するならA-4と接続

### A-4. オンラインランキング 【要・事前設計】
- Firestore活用。ただし**着手前に決めるべき設計判断が多い**：
  - **不正対策**：クライアントスコアは自己申告になり得る。改ざん耐性（サーバ側検証 or リプレイ検証 or 統計的外れ値排除）をどうするか
  - **セキュリティルール**：現状は書き込み専用。読み取り解禁時のFirestoreルール設計（読み取り範囲・レート制限）
  - **表示名のモデレーション**：不適切な名前の扱い
  - **プライバシー**：現在は匿名。名前を出すなら同意フローの見直し
- 一番重い。A-1〜A-3を先に。

### A-5. リプレイ（過去の対戦を再生）
- 対戦履歴には既にラウンド別の詳細（`rounds`：親子・出した札・勝敗・得点）を保存済み
- これを1手ずつ再生するビューを作れば実装可能。デイリー/共有とも相性良い

### A-6. AI解析「この場面の最善手」
- 既存の `cpuChooseCard` / `cpuEstimate`（期待値計算）を流用し、プレイヤーの各局面で最善手と実際の手を比較表示
- 初心者ヒント機能の発展形。学習価値が高い

## B. コード品質・保守性（レビュー指摘）

> 方針：機能追加を優先しつつ内部を軽く整理。大規模リファクタは急がない。
> 単一ファイルは**意図的な制約**（自己完結・ビルド不要・GitHub Pagesにそのまま配置・カード画像はbase64埋め込み）。

### B-1. ファイル分割（css/ js/ 等）
- ビルド無しでも `<link>` / `<script src>` で分割は可能（game.js / cpu.js / ui.js / animation.js / tutorial.js / firebase.js / images.js(base64) など）
- ただし現行400KB超の稼働ファイルを割るのは相応にリスク。分割するなら各機能の verify を用意してから段階的に
- base64画像を `images.js` に隔離するだけでも編集時の見通しは大きく改善する

### B-2. CSSの共通化
- `.card` 系（`.card` / `.faceup` / `.facedown` / `.art` 等）に重複が多い。共通プロパティを括り出して短縮

### B-3. アニメーション負荷（旧Android）
- glow / shadow / blur / transform が多く、古い端末で重くなる可能性
- 対策案：`prefers-reduced-motion` 対応、設定で「演出を控えめに」トグル、blur/box-shadow の多用箇所を軽量化

### B-4. JSの責務分離
- ルール / UI / CPU / Firebase / 保存 / 演出 が同居。B-1と合わせて責務ごとにモジュール化

## C. ゲームバランス（保留中）

### C-1. 数字5が弱い問題（体感の直感違反）
- 数字5の決戦勝率が約27%と最弱（大きい札＝強い、の直感に反する）
- **配り方では解決しない**＝ルール由来（Kaosの0/1が5に勝つ特殊ルール）。解決には看板ルールの変更（例：A3「特殊効果を撤廃し数字比較」）が必要で、代わりにKaos5が新たな強者になり「低札で大札を食う」心理戦が消えるトレードオフ
- 選択肢：(a) 現状維持＋ヒントで意図を伝える / (b) ルール変更を検討
- シミュレータ（scratchpadの `sim.js`）に検証モードあり（`kaostest` 等）

## メモ
- バランス検証はシミュレータ `sim.js`（本物のCPUロジック移植）で実施。ランダムbotではなく実ロジックで測ること
- 変更は毎回、実機（Playwright headless）で JSエラー0・通し完走を確認してからデプロイ
- デプロイは main へ push → GitHub Pages（pages build and deployment）で自動反映
